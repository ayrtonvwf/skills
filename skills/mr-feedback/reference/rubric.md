# MR Review-Readiness Rubric

This rubric measures **how ready an MR is to be reviewed** - is the context there,
is the scope clear, can a reviewer pick it up efficiently. It does **not** judge the
correctness or quality of the code itself. Never suggest code changes.

Premise to keep in tone: a well-prepared MR signals care. Reviewers prioritize MRs
that are easy to pick up, so readiness is not bureaucracy - it gets the MR reviewed
faster and better.

## The code-critique boundary

Noting that a readiness signal exists is in scope; instructing how to change the code
is not. You may **point out that** hygiene markers (leftover TODOs, debug logs,
commented-out blocks) are present in the diff, because their presence is a
self-review/readiness signal. You may **not** tell the author how to edit the code -
no "remove line X", no "rename this", no "use Y instead". Keep it to "this is here,
resolving it before review will help", never "here is the change to make".

- Allowed: "There are leftover debug/TODO markers in the diff (e.g. a `console.log` and
  a `// TODO`) - calling them out or clearing them before review will help."
- Disallowed: "Remove the `console.log("charging", amount)` and the
  `// TODO: clean this up` from `payment.ts`."
- Allowed: "The diff mixes a refactor with the feature; a note on what to review first
  would help."
- Disallowed: "Split this function" / "rename `foo` to `bar`" / "this should use a map."

## Categories

Score each 0-10, then combine into an overall score (weighted; high-weight gaps
drag the total down more than low-weight ones).

Each category has a **Reason** - a short explanation of why the rule matters to the
author. When a category surfaces a gap in the report, include a *why it matters* so
the author understands why closing the gap helps, not just what to fix. Use the Reason
as background only - rewrite it to match the exact problem found, and never reference
parts of the category that are actually fine (e.g. if the pipeline is green, don't
mention the pipeline when flagging a stale branch).

### 0. Required readiness items (weight: high)

These three are explicit must-haves for every MR. They are **not absolute gates** -
an MR missing one can still be scored - but each missing item sharply pulls the
overall score down and must always surface as an `[important]` item in the report.

- **Jira task link** - the description links the Jira task the MR resolves.
  **Exception:** if the description explicitly states there is no Jira card for this
  change (e.g. an ad-hoc fix, chore, or task that was never ticketed), treat this item
  as satisfied - it must **not** lower the score or surface as a gap. A missing link
  with no explanation is still a gap.
- **Testing instructions** - clear, reproducible steps a reviewer can follow to
  test the change **themselves**.
- **Proof it was run** - a screenshot of the rendered UI (for frontend changes)
  **or** a log / output generated by the **actual code being run** (for backend
  changes). This shows the author actually ran and verified the change, not just
  wrote it. **A passing test run is not proof-of-run** - it only proves the tests
  executed, not that the real code path was exercised. Proof-of-run must come from
  running the code itself (the rendered component, or logs/output the code produced
  in a real run), not from the test suite. Test files in the diff are scored
  separately in section 3.
- Red flag: any of the three missing, or test output offered in place of real
  proof-of-run.
- **Reason:** these three give the reviewer the traceability (Jira), a way to verify
  the change themselves (testing instructions), and confidence it actually works
  (proof-of-run). Proof-of-run specifically must show the *real code* running -
  passing tests prove the tests ran, not that the feature works as the user will
  experience it. Without these the reviewer has to reconstruct context and re-derive
  how to test, which stalls the review before it even starts.

### 1. Description & context (weight: high)

- Explains **what** changed and **why** (not just a restated title).
- Calls out breaking changes, migrations, feature flags, rollout/rollback notes.
- Red flag: empty description, existing description by default template with obvious empty sections.
- **Reason:** the description is the reviewer's entry point. Without the what and why
  they have to infer intent from the diff, and undocumented breaking changes or
  other important facts can slip through and cause incidents after merge.

### 2. Scope & size (weight: high)

- Single, coherent purpose - not several unrelated changes bundled together.
- Reviewable size. Rough heuristic: < ~400 changed lines is comfortable;
  > ~1000 needs strong justification or should be split.
- Red flag: refactor + feature + formatting mixed in one MR with no note explaining it.
- **Reason:** small, single-purpose MRs are reviewed faster and more thoroughly.
  Large or mixed-scope changes overwhelm the reviewer, hide real changes among noise,
  and raise the chance bugs are missed or the review is rubber-stamped.

### 3. Test coverage signals (weight: high)

- Behavior changes are accompanied by test files in the diff.
- Judge **presence and acknowledgement**, not test quality.
- This is about test files in the diff; reviewer-facing testing instructions and
  proof-of-run are scored in section 0.
- **Not applicable when** the MR is not an implementation-code change (e.g. just
  documentation, config, copy, or asset changes). In that case this section does
  **not** apply: do **not** treat missing tests as a gap, do **not** lower the score
  for it, and mark the category "not applicable" rather than penalizing it.
- Red flag: logic change with zero test files touched and no testing notes.
- **Reason:** tests in the diff show the author thought about how the change can
  break and let the reviewer trust the behavior without manually exercising every path.
  Logic changes with no tests shift that burden onto the reviewer and onto production.

### 4. Reviewer guidance (weight: medium)

- Does the MR help the reviewer? Where to start, what to focus on, known
  trade-offs, open questions, out-of-scope items deliberately left out.
- Self-review done (no leftover TODOs, debug markers called out, draft status clear).
- **Reason:** pointing the reviewer to where to start and what to watch for saves them
  the time of mapping the change blindly, and catching your own leftovers first means
  the reviewer spends their attention on what actually matters.

### 5. CI & mergeability (weight: medium)

- Pipeline is green.
- No merge conflicts; required approvals/threads not blocking.
- Red flag: failing pipeline with no acknowledgement in the description/discussion.
- **Reason:** a green pipeline and a conflict-free branch tell the reviewer the MR is
  in a mergeable state worth their time. A red or unacknowledged pipeline makes them
  guess whether failures are known or real, and merge conflicts block the work entirely.

## Scoring

- Each category gets a 0-10 sub-score with a one-line reason.
- The **overall score (0-10)** is a weighted judgment, not a strict average -
  a high-weight gap (no context, no tests, unreviewable size) should pull it down
  noticeably.
- A missing **Required readiness item** (section 0 - Jira link, testing
  instructions, proof-of-run) is among the heaviest gaps: each one missing should
  visibly lower the overall score, and an MR missing all three should not land in
  "Ready". They are weighted heavily but are not absolute gates.
- Items that are **legitimately not applicable do not count as missing** and must not
  lower the score: a Jira link the description explicitly says doesn't exist, or test
  coverage on a non-implementation MR (docs/config/assets). Treat these as satisfied /
  not applicable, not as gaps.
- Map the overall score to a label:

| Score | Label | Meaning |
| ----- | ----- | ------- |
| 9-10 | Ready | Reviewer can pick it up immediately. |
| 7-8 | Almost ready | Solid; a couple of additions would help the reviewer. |
| 4-6 | Needs prep | Missing context/tests/scope clarity; readiness gaps to close first. |
| 0-3 | Not ready | No reviewable content, or no context at all. |

Be honest but encouraging. The goal is to help the author present an MR that gets
reviewed faster - not to gatekeep.
